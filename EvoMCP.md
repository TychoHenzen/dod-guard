Why evolution hasn't worked
AlphaEvolve/OpenEvolve succeed in a narrow regime that "solve my problem" doesn't live in:

Dense scalar fitness. Matrix-multiply op counts, packing density — smooth, machine-gradable, every candidate gets a meaningful score. Your fitness is probably "tests pass / DoD satisfied," which is binary and sparse. Evolution on a binary landscape degenerates into random search with extra bookkeeping.
Small evolvable unit. AlphaEvolve evolves a kernel or a single function inside EVOLVE-BLOCK markers, not a feature across a codebase. Mutation of a whole change-set is basically resampling.
Thousands of evaluations. Evolution's advantage only appears at scale. If your eval is "run the full test suite" at 30s each, you never reach the regime where population dynamics beat best-of-N.
Diversity maintenance. Without islands/MAP-Elites and without showing the mutator the elite programs and their scores in-prompt, a dumb model produces near-identical children. DeepSeek especially mode-collapses hard.

So evolution isn't wrong, it's mismatched: it's an optimizer, and most of your work items aren't optimization problems.
The token-scaling menu, in order of expected ROI for "dumb but cheap"

Best-of-N with your verifier. Simplest and most underrated. pass@k grows surprisingly fast with k even for weak models — the whole economic argument for DeepSeek is that N=20 DeepSeek attempts filtered by dod-guard costs less than 1 Claude attempt. The catch: samples must actually differ. Force diversity at the plan level, not the code level — sample N one-paragraph approaches first, dedupe (embedding distance or just an LLM judge saying "same approach"), then implement each. Blind temperature-based resampling of code gives you 20 copies of the same bug.
Parallel restarts × short repair chains. Pure best-of-N wastes the information in failures; pure sequential self-repair plateaus fast with dumb models (they loop on the same error). The hybrid — N independent attempts, each allowed 2–3 repair iterations fed by concrete test output — dominates both. Kill a lineage when it repeats a failure signature; that's your stuck-detector.
Cascade escalation. You've circled this before, and it's the highest-leverage piece: DeepSeek does the fanout, and "same failing assertion after 3 lineages" is the trigger to spend Claude tokens on exactly that node. Most of the cost lives in the easy 90%; most of the failure lives in the hard 10%.
MAKER-style decomposition with per-step voting — worth it only when steps are individually verifiable; otherwise the decomposition itself becomes the failure point, and dumb models are worst at decomposition. Let the expensive model decompose, the cheap model execute.
Tree search / MCTS over repair actions — real gains in papers, but the engineering cost is high and it mostly matters when the verifier is expensive. Skip for now.

The form of evolution that would actually work
Keep it, but reposition it as a tool for optimization subproblems, and fix the fitness landscape:

Densify fitness using dod-guard itself. You already built a tech-debt-score skill. A weighted composite — % of tests passing (run individually, not suite-pass/fail), branch coverage delta, mutation score, lint/DoD violations, perf numbers — turns your binary gate into a gradient. This is the single change most likely to make evolution stop flailing: partial credit is what lets a population climb.
Evolve marked regions, not files. EVOLVE-BLOCK semantics; everything outside is frozen scaffold with a stable interface the tests hit.
Diff-based mutation with elites in context. The prompt to the mutator contains: the current program, the top-k programs with scores, and the score history. AlphaEvolve's ablations show this in-context "gradient" matters more than population size. Grammar-constrained decoding (which you were already exploring) is genuinely useful here — forcing valid unified-diff output is where dumb models bleed the most evaluations.
Islands with periodic migration, and a MAP-Elites axis or two (e.g. length × perf) so you keep weird candidates alive.

Use it when the task sounds like "make this pass more property tests / run faster / score higher," and route "implement feature X" to the best-of-N + cascade path instead. One orchestrator, two strategies, dispatched by whether the fitness is scalar or binary.
MCP inside the dod-guard plugin: yes, and it's the right boundary
Claude Code plugins can bundle MCP servers via the plugin manifest (an mcpServers entry alongside your hooks/commands — check current docs for the exact schema, it's moved around across versions). The architectural reason to do it this way is context economy: if the agent does best-of-N in its own conversation, you pay Claude-context prices to shuttle DeepSeek candidates around and you pollute the working context with 19 failures. Instead:

MCP server exposes something like solve(task_spec, verify_cmd, budget_tokens, strategy) and evolve(target_region, fitness_cmd, generations).
The server internally owns the DeepSeek API fanout, the repair loops, the population state, and the escalation policy.
It returns only the winning patch plus the verification report (and maybe a one-line summary of what failed approaches taught it).
dod-guard's existing checks become the verify_cmd/fitness_cmd — the plugin is already the source of truth for "done," so co-locating the search engine with the judge keeps them from drifting apart.

The one thing I'd warn against: don't let the MCP server also be an agent with its own tool loop for file editing. Keep it a pure function from (task, repo snapshot, budget) → (patch, report). Determinism and replayability there will save you weeks of debugging the orchestration itself — which, given the theme, would be a fittingly ironic failure mode.

This is the step that genuinely needs a smart model, and it's a few thousand tokens instead of a hundred thousand.
2. Claude calls one MCP tool and goes quiet. Something like solve(spec, verify_cmd, budget). From Claude's perspective this is a single blocking tool call. All the token-burning happens inside the MCP server, invisible to the main context — that's the whole point of the boundary. Claude's context never sees 19 failed candidates.
3. The MCP server dispatches on one question: is the fitness binary or scalar? "Implement this endpoint" → tests pass or don't → best-of-N pipeline. "Make this function faster / raise mutation score" → there's a number to climb → evolution pipeline. You don't choose; the shape of the verification chooses.
4. The cheap engine grinds. Best-of-N path: DeepSeek samples N plans, dedupes them, implements each, and each lineage gets 2–3 repair attempts fed by real test output before being killed. Evolution path: population of diffs over a marked region, dod-guard's composite score as fitness, elites shown to the mutator. In both cases dod-guard is the judge running locally against every candidate — it stops being a gate at the end and becomes the selection pressure throughout.
5. One of two things comes back. Either a winning patch plus its verification report (Claude reviews it — one cheap read — and applies it), or an escalation report: "every lineage died on the same assertion, here's the failure signature and the best partial attempt." That report is precisely the input Claude needs to spend expensive tokens on the one hard sub-problem rather than the whole task. Claude either solves that node directly or rewrites the spec and re-invokes.
6. Your existing dod-guard hooks still gate the final commit. The solver using dod-guard as fitness doesn't replace the outer enforcement — it just means candidates arriving at the gate have already been selected to pass it.

The new loop is smart model defines success, dumb models search, verifier scores, smart model only touches what the search couldn't crack.
